01

O que é Context API e por que ela existe

ThemeContext.tsx 
const ThemeContext = createContext<ThemeContextValue | null>(null)

Context API é o mecanismo nativo do React para distribuir estado pela árvore de componentes sem passar props manualmente a cada nível. O problema que ela resolve tem nome: prop drilling — a prática de passar um dado por vários componentes intermediários que não precisam daquele dado, apenas para que um descendente distante o receba.

Context API em React puro vs. Next.js App Router

A diferença não está na API — createContext, useContext e useState funcionam da mesma forma. A diferença está no ambiente de execução.

Em React puro (Vite), todo código roda no browser. Não existe servidor. localStorage, window e document estão sempre disponíveis. No Next.js App Router, cada arquivo é um Server Component por padrão — executado no servidor, onde browser APIs não existem.

Ambiente Execução padrão Context API Browser APIs
React puro (Vite) Browser Disponível diretamente Sempre acessíveis
Next.js App Router Servidor Requer 'use client' Requer guard typeof window
⚠️

Sem a diretiva 'use client', o Next.js tenta executar o Provider no servidor e lança erro ao encontrar useState. O erro não é sobre Context em si — é sobre hooks rodando fora do ambiente de browser.

02

Modelagem dos tipos

ThemeContext.tsx 
type Theme = 'light' | 'dark'

type ThemeContextValue = {
  theme: Theme
  toggleTheme: () => void
}
Linha O que faz
type Theme = 'light' | 'dark' Union type literal. O TypeScript aceita apenas essas duas strings. Qualquer outro valor causa erro em tempo de compilação.
type ThemeContextValue Define o contrato do contexto — o que qualquer consumidor vai receber ao chamar useTheme.
theme: Theme O estado atual. Leitura apenas — o consumidor não modifica diretamente.
toggleTheme: () => void A única ação disponível. O consumidor chama a função sem saber como o estado muda internamente.

O createContext recebe ThemeContextValue | null porque no momento da criação o contexto não tem valor. O null é o estado inicial antes de qualquer Provider existir na árvore. O useTheme trata esse caso com um guard explícito.

03

A função auxiliar applyTheme

ThemeContext.tsx 
const applyTheme = (theme: Theme): void => {
  document.documentElement.setAttribute('data-theme', theme)
  localStorage.setItem('theme', theme)
}
Linha O que faz
document.documentElement Referência ao elemento <html>. É onde o atributo data-theme precisa estar para que o @custom-variant dark do Tailwind funcione em todos os descendentes.
.setAttribute('data-theme', theme) Escreve data-theme="light" ou data-theme="dark" no <html>. O Tailwind lê esse atributo via a custom variant declarada no CSS.
localStorage.setItem('theme', theme) Persiste a preferência. Na próxima visita, resolveInitialTheme vai encontrar esse valor e restaurar o tema sem flash.
💡

Essa função é declarada fora do componente porque não depende de estado ou props. Função pura de efeito colateral: recebe um valor, escreve em dois lugares externos. Declarar fora evita recriação a cada render.

04

resolveInitialTheme e o guard de ambiente

ThemeContext.tsx 
const resolveInitialTheme = (): Theme => {
  if (typeof window === 'undefined') return 'light'

  const stored = localStorage.getItem('theme')
  if (stored === 'light' || stored === 'dark') return stored
  return window.matchMedia('(prefers-color-scheme: dark)').matches
    ? 'dark'
    : 'light'
}
Linha O que faz
typeof window === 'undefined' Guard de ambiente. No servidor, window não existe — acessá-lo lança ReferenceError. Retorna 'light' como fallback seguro antes de qualquer código browser rodar.
localStorage.getItem('theme') Lê a preferência persistida. Retorna null se não existir.
stored === 'light' || stored === 'dark' Validação explícita. O TypeScript precisa dessa checagem para aceitar stored como Theme — string | null não é Theme.
window.matchMedia(...).matches Lê a preferência do sistema operacional. Se o usuário nunca visitou o site, o tema inicial segue o SO.

A hierarquia de decisão é: storage › preferência do sistema › fallback light.

Inicializador lazy do useState

✅ Correto
useState(resolveInitialTheme)
// React chama uma vez na montagem
❌ Incorreto
useState(resolveInitialTheme())
// Executa durante o render
// ReferenceError no SSR

Passar a referência (sem parênteses) é o padrão de inicializador lazy do useState. O React garante que a função seja chamada apenas uma vez, no cliente, após a montagem.

05

O Provider

src/contexts/ThemeContext.tsx 
'use client'

import { createContext, useContext, useState } from 'react'

type Theme = 'light' | 'dark'

type ThemeContextValue = {
  theme: Theme
  toggleTheme: () => void
}

const ThemeContext = createContext<ThemeContextValue | null>(null)

type ThemeProviderProps = { children: React.ReactNode }

const applyTheme = (theme: Theme): void => {
  document.documentElement.setAttribute('data-theme', theme)
  localStorage.setItem('theme', theme)
}

const resolveInitialTheme = (): Theme => {
  if (typeof window === 'undefined') return 'light'
  const stored = localStorage.getItem('theme')
  if (stored === 'light' || stored === 'dark') return stored
  return window.matchMedia('(prefers-color-scheme: dark)').matches ? 'dark' : 'light'
}

export const ThemeProvider = ({ children }: ThemeProviderProps) => {
  const [theme, setTheme] = useState<Theme>(resolveInitialTheme)

  const toggleTheme = (): void => {
    const next: Theme = theme === 'light' ? 'dark' : 'light'
    applyTheme(next)
    setTheme(next)
  }

  return <ThemeContext value={{ theme, toggleTheme }}>{children}</ThemeContext>
}
Linha O que faz
useState<Theme>(resolveInitialTheme) Inicializa o estado com o tema resolvido. O React chama resolveInitialTheme uma vez na montagem no cliente.
const next: Theme Calcula o próximo tema antes de qualquer escrita. Evita repetir a expressão em applyTheme e setTheme.
applyTheme(next) Escreve no DOM e no storage antes de setar o estado React. O browser aplica a mudança visual imediatamente, sem esperar o próximo render.
setTheme(next) Atualiza o estado React para sincronizar a UI — o label do botão precisa refletir o tema atual.
<ThemeContext value={...}> Sintaxe do React 19. Em versões anteriores era <ThemeContext.Provider value={...}>.
⚠️

A ordem importa. Se setTheme viesse antes de applyTheme, o React agendaria um re-render com o novo estado — mas o data-theme no <html> ainda seria o antigo. O resultado seria um frame com estado React novo e DOM antigo. Chamando applyTheme primeiro, o DOM muda sincronicamente.

06

O hook useTheme

src/contexts/ThemeContext.tsx 
export const useTheme = (): ThemeContextValue => {
  const ctx = useContext(ThemeContext)
  if (!ctx) throw new Error('useTheme deve ser usado dentro de ThemeProvider')
  return ctx
}
Linha O que faz
useContext(ThemeContext) Lê o valor do Provider mais próximo na árvore. Se não houver Provider, retorna o valor inicial do createContext — que é null.
if (!ctx) throw new Error(...) Guard contra uso fora do Provider. Transforma erro silencioso (contexto null) em mensagem clara de diagnóstico.
return ctx Após o guard, o TypeScript sabe que ctx não é null. O tipo é ThemeContextValue, sem necessidade de non-null assertion.
💡

Esse padrão de hook wrapper é a convenção para contextos com TypeScript. Sem ele, todo consumidor precisaria checar null manualmente — ou usar o operador ! (non-null assertion), que mascara o problema sem resolver.

07

O componente ThemeToggle

src/components/ThemeToggle/ThemeToggle.tsx 
'use client'

import { Moon, Sun } from 'lucide-react'
import { useTheme } from '@/contexts/ThemeContext'

export const ThemeToggle = () => {
  const { theme, toggleTheme } = useTheme()
  const isDark = theme === 'dark'

  return (
    <button
      onClick={toggleTheme}
      aria-label={isDark ? 'Ativar modo claro' : 'Ativar modo escuro'}
      className="flex items-center gap-2 px-4 py-2 rounded-lg
        border border-black/10 bg-white text-gray-800
        hover:bg-gray-100
        dark:border-white/10 dark:bg-zinc-800
        dark:text-gray-100 dark:hover:bg-zinc-700
        transition-colors duration-200 cursor-pointer"
    >
      {isDark ? <Sun size={18} /> : <Moon size={18} />}
      <span className="text-sm font-medium">
        {isDark ? 'Modo claro' : 'Modo escuro'}
      </span>
    </button>
  )
}
Linha O que faz
'use client' Obrigatório. useTheme usa useContext — hook exclusivo de Client Components.
const isDark = theme === 'dark' Estado derivado local. Evita repetir theme === 'dark' em três lugares distintos no JSX.
aria-label={isDark ? ...} Acessibilidade. O label descreve a ação que o botão executa, não o estado atual. "Ativar modo claro" quando o tema é dark é mais informativo que "Tema: dark".
{isDark ? <Sun /> : <Moon />} O ícone reflete o que o clique vai fazer — Sol indica que clicar ativa o claro, Lua indica escuro.
08

Integração no layout.tsx

src/app/layout.tsx 
import { ThemeProvider } from '@/contexts/ThemeContext'

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="pt-BR" suppressHydrationWarning>
      <body>
        <ThemeProvider>
          {children}
        </ThemeProvider>
      </body>
    </html>
  )
}
Linha O que faz
suppressHydrationWarning Instrui o React a ignorar divergências de atributos no <html>. O data-theme gerado no servidor ("light") diverge do valor lido do localStorage no cliente — esse atributo é o único ponto de divergência esperada.
<ThemeProvider> envolvendo {children} Posiciona o Provider acima de toda a árvore. Qualquer componente em qualquer rota pode chamar useTheme.
ℹ️

suppressHydrationWarning não é um hack. Age apenas no elemento <html> e em seus atributos diretos. O data-theme vive exatamente ali. O resto da árvore continua sendo verificado normalmente pelo React.

09

A @custom-variant do Tailwind v4

src/app/globals.css 
@import "tailwindcss";

@custom-variant dark (&:where([data-theme=dark], [data-theme=dark] *));

html {
  transition: background-color 0.3s ease, color 0.3s ease;
}

O Tailwind v4 por padrão detecta tema via classe .dark no <html>. Essa linha substitui esse comportamento por detecção via atributo data-theme.

O seletor &:where([data-theme=dark], [data-theme=dark] *) funciona assim:

Parte do seletor O que faz
[data-theme=dark] Aplica no próprio elemento que tem o atributo.
[data-theme=dark] * Aplica em todos os descendentes desse elemento.
:where(...) Usa especificidade zero, para não sobrescrever estilos com especificidade normal.

Com essa variant, dark:bg-zinc-800 compila para:

CSS compilado 
:where([data-theme=dark], [data-theme=dark] *) .dark\:bg-zinc-800 {
  background-color: rgb(39 39 42);
}
10

Fluxo completo da troca de tema

Ao clicar no botão

Usuário clica no botão
→ onClick dispara toggleTheme
→ next = theme === 'light' ? 'dark' : 'light'
→ applyTheme(next)
→ document.documentElement.setAttribute('data-theme', next) ← DOM atualiza
→ localStorage.setItem('theme', next) ← storage persiste
→ setTheme(next)
→ React agenda re-render
→ ThemeToggle re-renderiza com isDark atualizado
→ ícone e label mudam
→ CSS transition suaviza background e color

Na próxima visita

Browser carrega a página
→ Next.js renderiza no servidor com theme = 'light' (fallback)
→ HTML chega ao browser com data-theme="light"
→ React hidrata
→ ThemeProvider monta no cliente
→ useState chama resolveInitialTheme()
→ localStorage.getItem('theme') → 'dark'
→ applyTheme('dark') ← data-theme="dark" aplicado
→ suppressHydrationWarning ignora a divergência no <html>
—

Resumo das decisões de design

Decisão Justificativa
createContext<T | null>(null) O null inicial força o guard no useTheme. Sem ele, o TypeScript aceitaria undefined silenciosamente e os erros apareceriam em runtime, não em compilação.
applyTheme fora do componente Função pura sem dependências de estado ou props. Declarar dentro do componente a recriaria a cada render sem necessidade.
typeof window === 'undefined' Guard de ambiente que permite o Provider ser carregado sem lançar ReferenceError durante o SSR, mesmo com 'use client'.
useState(resolveInitialTheme) sem parênteses Inicializador lazy: o React chama a função uma vez na montagem. Com parênteses, executaria durante o render — antes do guard de ambiente ter efeito.
applyTheme antes de setTheme Garante que o DOM mude sincronicamente antes do React agendar o re-render. Evita um frame com estado React novo e DOM antigo.
suppressHydrationWarning no <html> Escopo preciso: age apenas no <html> e seus atributos diretos. O data-theme diverge deliberadamente entre servidor e cliente — não indica bug.
Hook useTheme wrapper Transforma erro silencioso (contexto null) em mensagem clara. Elimina checagem manual de null em cada consumidor.
@custom-variant dark com data-theme Controle programático via setAttribute. A variante padrão do Tailwind exige manipulação de classList — o atributo é mais direto e legível.

Exercícios

Três exercícios em progressão deliberada. O básico é reprodução dirigida da técnica. O médio exige extensão da implementação existente com decisão estrutural. O avançado exige composição de múltiplas técnicas com TypeScript genérico.

Básico Exercício 1 — FontSizeContext

Crie um contexto FontSizeContext que disponibilize três tamanhos de fonte: 'sm', 'md' e 'lg'. Implemente um Provider com useState, um hook useFontSize com guard de null, e um componente FontSizeControl com três botões que permitem ao usuário trocar o tamanho.

  • Aplique o valor como atributo data-font-size no document.documentElement
  • O hook deve lançar Error se usado fora do Provider
  • Inclua a diretiva 'use client' nos arquivos necessários
Critério de aceitação: clicar em cada botão altera o atributo no <html> sem erro de TypeScript. O hook lança Error descritivo se usado fora do Provider.
Solução — FontSizeContext.tsx 
'use client'

import { createContext, useContext, useState } from 'react'

type FontSize = 'sm' | 'md' | 'lg'

type FontSizeContextValue = {
  fontSize: FontSize
  setFontSize: (size: FontSize) => void
}

const FontSizeContext = createContext<FontSizeContextValue | null>(null)

const applyFontSize = (size: FontSize): void => {
  document.documentElement.setAttribute('data-font-size', size)
}

export const FontSizeProvider = ({ children }: { children: React.ReactNode }) => {
  const [fontSize, setFontSizeState] = useState<FontSize>('md')

  const setFontSize = (size: FontSize): void => {
    applyFontSize(size)
    setFontSizeState(size)
  }

  return (
    <FontSizeContext value={{ fontSize, setFontSize }}>
      {children}
    </FontSizeContext>
  )
}

export const useFontSize = (): FontSizeContextValue => {
  const ctx = useContext(FontSizeContext)
  if (!ctx) throw new Error('useFontSize deve ser usado dentro de FontSizeProvider')
  return ctx
}

// ─── FontSizeControl.tsx ────────────────────────────────
export const FontSizeControl = () => {
  const { fontSize, setFontSize } = useFontSize()
  const sizes: FontSize[] = ['sm', 'md', 'lg']

  return (
    <div className="flex gap-2">
      {sizes.map(size => (
        <button
          key={size}
          onClick={() => setFontSize(size)}
          aria-pressed={fontSize === size}
        >
          {size.toUpperCase()}
        </button>
      ))}
    </div>
  )
}
Médio Exercício 2 — Tema com modo 'system'

Estenda o ThemeContext existente para suportar um terceiro tema: 'system'. O tema 'system' não aplica nenhum data-theme fixo — ele observa window.matchMedia('(prefers-color-scheme: dark)') e aplica 'dark' ou 'light' dinamicamente conforme a preferência do sistema muda.

  • Implemente o listener com addEventListener('change', ...)
  • Faça a limpeza com removeEventListener no cleanup do useEffect
  • O ThemeToggle deve exibir um terceiro estado para o tema 'system'
  • O tipo Theme deve ser estendido para 'light' | 'dark' | 'system'
Critério de aceitação: com tema 'system' ativo, mudar a preferência do SO no painel do Chrome DevTools (Rendering → Emulate CSS prefers-color-scheme) reflete a mudança visual na aplicação sem clique do usuário.
Solução — ThemeContext com 'system' 
'use client'

import { createContext, useContext, useEffect, useState } from 'react'

type Theme = 'light' | 'dark' | 'system'

type ThemeContextValue = {
  theme: Theme
  setTheme: (t: Theme) => void
}

const ThemeContext = createContext<ThemeContextValue | null>(null)

const applyResolved = (resolved: 'light' | 'dark'): void => {
  document.documentElement.setAttribute('data-theme', resolved)
}

export const ThemeProvider = ({ children }: { children: React.ReactNode }) => {
  const [theme, setThemeState] = useState<Theme>(() => {
    if (typeof window === 'undefined') return 'light'
    const stored = localStorage.getItem('theme')
    if (stored === 'light' || stored === 'dark' || stored === 'system') return stored
    return 'system'
  })

  useEffect(() => {
    if (theme !== 'system') {
      applyResolved(theme)
      return
    }

    const mq = window.matchMedia('(prefers-color-scheme: dark)')
    applyResolved(mq.matches ? 'dark' : 'light')

    const handler = (e: MediaQueryListEvent) =>
      applyResolved(e.matches ? 'dark' : 'light')

    mq.addEventListener('change', handler)
    return () => mq.removeEventListener('change', handler)
  }, [theme])

  const setTheme = (t: Theme): void => {
    localStorage.setItem('theme', t)
    setThemeState(t)
  }

  return <ThemeContext value={{ theme, setTheme }}>{children}</ThemeContext>
}

export const useTheme = (): ThemeContextValue => {
  const ctx = useContext(ThemeContext)
  if (!ctx) throw new Error('useTheme deve ser usado dentro de ThemeProvider')
  return ctx
}

// ─── ThemeToggle com três estados ───────────────────────
export const ThemeToggle = () => {
  const { theme, setTheme } = useTheme()
  const options: { value: Theme; label: string }[] = [
    { value: 'light',  label: '☀️ Claro'  },
    { value: 'dark',   label: '🌙 Escuro' },
    { value: 'system', label: '💻 Sistema' },
  ]
  return (
    <div className="flex gap-1 rounded-lg border border-black/10 p-1">
      {options.map(opt => (
        <button
          key={opt.value}
          onClick={() => setTheme(opt.value)}
          aria-pressed={theme === opt.value}
          className={theme === opt.value ? 'bg-white dark:bg-zinc-700' : ''}
        >
          {opt.label}
        </button>
      ))}
    </div>
  )
}
Avançado Exercício 3 — UserPreferencesContext

Implemente um UserPreferencesContext que gerencie três preferências simultaneamente: theme ('light' | 'dark'), language ('pt-BR' | 'en') e reducedMotion (boolean). O estado deve ser persistido em localStorage como um único objeto JSON.

  • O Provider deve ler, validar e restaurar o objeto na montagem — com fallback se o JSON estiver malformado ou ausente
  • Crie um hook useUserPreferences que exponha o estado completo e uma função genérica: updatePreference<K extends keyof Preferences>(key: K, value: Preferences[K]): void
  • Aplique reducedMotion como atributo data-reduced-motion no <html>
  • No globals.css: [data-reduced-motion=true] * { transition: none !important }
Critério de aceitação: updatePreference('theme', 'dark') não sobrescreve language ou reducedMotion no storage. O TypeScript rejeita em compilação chamadas com chave inexistente ou valor incompatível com a chave.
Solução — UserPreferencesContext.tsx 
'use client'

import { createContext, useContext, useState } from 'react'

type Preferences = {
  theme:        'light' | 'dark'
  language:     'pt-BR' | 'en'
  reducedMotion: boolean
}

type PreferencesContextValue = {
  preferences: Preferences
  updatePreference: <K extends keyof Preferences>(key: K, value: Preferences[K]) => void
}

const DEFAULTS: Preferences = {
  theme:         'light',
  language:      'pt-BR',
  reducedMotion: false,
}

const STORAGE_KEY = 'user-preferences'

const loadPreferences = (): Preferences => {
  if (typeof window === 'undefined') return { ...DEFAULTS }
  try {
    const raw = localStorage.getItem(STORAGE_KEY)
    if (!raw) return { ...DEFAULTS }
    const parsed = JSON.parse(raw) as Partial<Preferences>
    return { ...DEFAULTS, ...parsed }
  } catch {
    return { ...DEFAULTS }
  }
}

const applyPreferences = (prefs: Preferences): void => {
  const root = document.documentElement
  root.setAttribute('data-theme', prefs.theme)
  root.setAttribute('data-reduced-motion', String(prefs.reducedMotion))
  localStorage.setItem(STORAGE_KEY, JSON.stringify(prefs))
}

const PreferencesContext = createContext<PreferencesContextValue | null>(null)

export const UserPreferencesProvider = ({ children }: { children: React.ReactNode }) => {
  const [preferences, setPreferences] = useState<Preferences>(loadPreferences)

  const updatePreference = <K extends keyof Preferences>(
    key: K,
    value: Preferences[K]
  ): void => {
    const next = { ...preferences, [key]: value }
    applyPreferences(next)
    setPreferences(next)
  }

  return (
    <PreferencesContext value={{ preferences, updatePreference }}>
      {children}
    </PreferencesContext>
  )
}

export const useUserPreferences = (): PreferencesContextValue => {
  const ctx = useContext(PreferencesContext)
  if (!ctx) throw new Error('useUserPreferences fora do UserPreferencesProvider')
  return ctx
}
